Level 5 レッスン 5-7

ベイズワークフロー — モデル構築から検証まで

このレッスンでできるようになること

  • ベイズ分析を「事前予測チェック → 推定 → 収束確認 → 事後予測チェック → モデル批判 → 反復」という一連の流れとして説明できる
  • 事前予測チェックと事後予測チェックの違いを、何を見て何を判断するかという観点で区別できる
  • 1つのモデルで終わらせず、批判して作り直すという反復の姿勢を自分の言葉で語れる

前提: 5-1 PyMCではじめるベイズ推定・4-6 収束診断とサンプリングの実務・3-7 モデル比較とベイズファクター を読んでいること

「モデルを書いて、はい完成」ではない

ここまでの旅で、たくさんの道具を手に入れました。ベイズの定理、事前分布と事後分布、尤度（ゆうど）、MCMC、PyMCでの回帰や階層モデル。最後のレッスンでは、それらをどういう順番で、どう組み合わせて使うかを学びます。

現実のデータ分析では、いきなり完璧なモデルは書けません。最初のモデルはたいてい、どこか間違っています。事前分布が変だったり、データの形に合っていなかったり。大切なのは、その間違いに自分で気づいて直すことです。

この「書いて、確かめて、直す」を一連の手順にまとめたものをベイズワークフロー（Bayesian workflow）と呼びます。料理のレシピのように、迷ったときに立ち返れる地図です。今日はこの地図を、サイト全体のまとめとして広げてみましょう。

直感でつかむ

家を建てることを思い浮かべてください。設計図をいきなり完成させて、そのまま建てる大工さんはいません。まず模型を作って「玄関の向き、これでいい?」と確かめます（事前予測チェック）。建てている途中も「柱はまっすぐ立っているか」と点検します（収束確認）。建て終わったら「雨漏りしないか、住み心地はどうか」を実際に試します（事後予測チェック）。問題が見つかれば設計を直して、また建て直す。ベイズワークフローも、これとそっくりの「点検しながら作る」やり方なのです。

ワークフロー全体の地図

まず全体像を一枚の図で見ましょう。ベイズ分析は、次の流れをぐるぐる回るサイクルです。一直線ではなく、後ろに戻る矢印があるのがポイントです。

それぞれのステップを、順番に見ていきましょう。①「モデルを書く」と③「推定する」はすでに 5-1 以降で何度もやってきました。今日の主役は、②の事前予測チェックと⑤の事後予測チェック、そして⑥のモデル批判です。

② 事前予測チェック — 推定の前に事前分布を点検する

モデルを書いたら、すぐにデータを当てはめたくなります。でも、その前に一歩立ち止まります。事前予測チェック（じぜんよそくチェック / prior predictive check）です。これは「データをまだ見ないで、事前分布だけからどんなデータが生まれるかを眺める」点検です。

定義 — 事前予測分布

パラメータ \(\theta\)（シータ）の事前分布 \(p(\theta)\) と、そのもとでのデータの分布 \(p(y \mid \theta)\) を組み合わせると、データ \(\tilde{y}\)（ワイ・チルダ、「まだ見ぬ仮のデータ」の意味）が生まれる分布が決まります。これを事前予測分布と呼びます: $$ p(\tilde{y}) = \int p(\tilde{y} \mid \theta)\, p(\theta)\, d\theta $$ 記号 \(\int \cdots\, d\theta\) は「\(\theta\) のあらゆる値について、その確率で重みづけて足し合わせる」という意味です（付録a-2）。

式は難しそうですが、やることは単純です。事前分布から \(\theta\) を1つサンプルし、その \(\theta\) を使ってデータを1セット生成する。これを何度も繰り返して、出てきた仮データを眺めるだけです。手順にすると次の通りです。

手順 — 事前予測チェック

\(s = 1, 2, \dots, S\) について繰り返す:

$$ \theta^{(s)} \sim p(\theta), \qquad \tilde{y}^{(s)} \sim p\!\left(\tilde{y} \mid \theta^{(s)}\right) $$

こうして得た仮データ \(\tilde{y}^{(1)}, \dots, \tilde{y}^{(S)}\) が、常識的にありえる範囲に収まっているかを目で確かめます。

なぜこれが大事なのでしょう。たとえば人の身長を予測するモデルで、事前分布がいいかげんだと、事前予測分布が「身長マイナス3メートル」や「身長500メートル」を平気で生み出すことがあります。データを見る前に、その時点で事前分布が間違っているとわかるのです。

つまずき注意 — 「無情報なつもり」が暴れる

「何も決めつけたくないから、うんと広い事前分布にしよう」と \(\theta \sim \mathcal{N}(0, 1000^2)\) のような極端に幅広い設定をすると、事前予測分布がとんでもない値ばかり出すことがあります。広ければ中立、というわけではありません。事前予測チェックは、こうした「無情報なつもりで実は暴れている事前分布」をあぶり出してくれます。多くの場合、ほどよく弱い情報を持たせた事前分布（弱情報事前分布）のほうが安全です。

例題 — 事前予測チェックでおかしさに気づく

あるコインの表が出る確率 \(\theta\) を推定したいとします。あなたは「\(\theta\) は0から1の間」と知っていますが、うっかり事前分布を \(\theta \sim \mathcal{N}(0.5,\ 0.4^2)\)（平均0.5、標準偏差0.4の正規分布）にしてしまいました。事前予測チェックで何が問題だとわかりますか。

解:

事前分布から \(\theta\) をサンプルしてみます。標準偏差が \(0.4\) と大きいので、\(\theta = -0.3\) や \(\theta = 1.2\) のような値が普通に出てしまいます。確率なのに0未満や1超えになるのは明らかにおかしい。データを見るまでもなく、この事前分布は不適切だと判明します。

正しくは、確率を表すなら値が必ず0〜1に収まるベータ分布、たとえば \(\theta \sim \text{Beta}(2, 2)\) を使うべきでした。事前予測チェックは「使ってはいけない事前分布」を、推定を回す前に止めてくれます。

④ 収束を確認する — サンプリングを信じてよいか

事前分布が妥当だと確認できたら、いよいよ③でMCMCを回して事後分布のサンプルを得ます。でも、得られたサンプルをすぐ信用してはいけません。MCMCがちゃんと事後分布に行き着いたかを確かめる必要があります。これが④の収束確認です。

くわしくは 4-6 収束診断とサンプリングの実務 で学びました。ここでは要点だけ思い出しておきましょう。チェックすべきは主に次の3つです。

見るもの意味合格の目安
\(\hat{R}\)(アール・ハット)複数の鎖が同じ分布に集まったかすべて 1.01 以下
有効サンプルサイズ実質いくつ分の独立サンプルか数百以上あれば安心
トレースプロット鎖の軌跡を時系列で見た図毛虫のように一様に混ざる

もし \(\hat{R}\) が大きかったり、警告が出たりしたら、その推定結果は使えません。サンプル数を増やす、モデルを書き直す、といった対処に戻ります。ここでも「先に進む前に点検」という姿勢は同じです。

⑤ 事後予測チェック — モデルは現実のデータを再現できるか

収束も確認できました。事後分布が手に入りました。では、そのモデルは良いモデルでしょうか。それを確かめるのが、ワークフローでいちばん大切といってもいい事後予測チェック（じごよそくチェック / posterior predictive check）です。

定義 — 事後予測分布

データ \(y\) を見たあとの事後分布 \(p(\theta \mid y)\) を使って、新しいデータ \(\tilde{y}\) が生まれる分布を事後予測分布と呼びます: $$ p(\tilde{y} \mid y) = \int p(\tilde{y} \mid \theta)\, p(\theta \mid y)\, d\theta $$ 事前予測分布の式と、ほとんど同じ形です。ちがいは事前分布 \(p(\theta)\) が事後分布 \(p(\theta \mid y)\) に置き換わっている一点だけです。

直感でつかむ — 二つの予測チェックの違い

事前予測チェックは「データを見る前に、思い込み（事前分布）が暴れていないか」を見ます。事後予測チェックは「データを見た後に、学んだモデルが本物のデータそっくりの偽データを作れるか」を見ます。

たとえるなら、事前予測チェックは試験前に「この勉強の方針、的外れじゃないか」を確認すること。事後予測チェックは試験後に「学んだ知識で、本番と同じ問題を解いて再現できるか」を試すこと。同じ「予測を眺める」でも、見る目的とタイミングが違うのです。

やり方はこうです。事後分布からモデルが作った偽データ \(\tilde{y}\) を、何セットも生成します。そして本物のデータ \(y\) と並べて見比べるのです。偽データの分布の中に本物がうまく収まっていれば、モデルは現実をよく再現できています。本物だけが偽データの群れから外れていたら、モデルはどこかを取りこぼしています。

つまずき注意 — 「本物のデータで答え合わせ」と勘違いしない

事後予測チェックは「予測が当たったかを採点する」テストではありません。見ているのは「モデルが、観測データと同じような特徴のデータを生み出せるか」です。たとえば本物のデータには外れ値が時々あるのに、モデルの偽データには一切出ないなら、そのモデルは「ばらつきの大きさ」を取りこぼしています。値を当てる精度ではなく、データの形・性質を再現できているかに注目するのがコツです。

⑥ モデル批判と比較 — そして反復へ

事後予測チェックで「ここがズレている」と見つかったら、何が原因かを考えます。これがモデル批判（model criticism）です。批判というと厳しく聞こえますが、要は「このモデルの弱点はどこか」を冷静に探す作業です。

原因の候補はいろいろあります。分布の選び方が合っていない（左右対称な正規分布を使ったが、本当は右に裾を引いていた、など）。説明に使う変数が足りない。グループ差を無視している（それなら階層モデルの出番です）。見つけた弱点をもとにモデルを書き直し、図1の点線をたどって①へ戻ります。これが⑦の反復です。

候補が複数あるときは、どれが良いかを比べます。3-7 モデル比較とベイズファクター で学んだ考え方の出番です。「データへの当てはまり」と「モデルの複雑さ」のバランスで選ぶ、というのが基本の発想でした。むやみに複雑なモデルが偉いわけではなく、シンプルで現実をよく説明できるモデルを目指します。

やってみよう

下のコードは、PyMCでの事前予測チェックと事後予測チェックの呼び出し方です。前のレッスンで書いたモデルに、たった数行を足すだけで点検ができます。sample_prior_predictive が②、sample_posterior_predictive が⑤に対応します。手元に環境がある人は、自分のモデルでぜひ試してみてください。

import pymc as pm
import arviz as az

with pm.Model() as model:
    theta = pm.Beta("theta", alpha=2, beta=2)        # 事前分布(0〜1に収まる)
    y = pm.Binomial("y", n=100, p=theta, observed=62)

    # ② 事前予測チェック: データを見る前に事前分布を点検
    prior_pred = pm.sample_prior_predictive()
    az.plot_ppc(prior_pred, group="prior")           # 事前予測分布を図示

    # ③ 推定 + ④ 収束確認
    idata = pm.sample()                              # MCMC実行
    az.summary(idata)                                # R-hat などを表示

    # ⑤ 事後予測チェック: 学んだモデルが本物のデータを再現できるか
    idata.extend(pm.sample_posterior_predictive(idata))
    az.plot_ppc(idata)                               # 偽データと本物を重ねて図示

理解度チェック

問1: 事前予測チェックは、ワークフローのどの時点で行いますか?

事前予測チェックは「データをまだ見ないで、事前分布だけからどんなデータが生まれるか」を点検します。だから推定（MCMC）を回す前に行います。図1の②の位置です。これにより、暴れた事前分布を早い段階で止められます。

問2: 事後予測チェックで見ているのは、次のうちどれですか?

事後予測チェックは、事後分布から偽データを何セットも生成し、本物のデータと見比べる作業です。本物が偽データの群れの中にうまく収まっていれば合格。aは事前予測チェック、bは収束確認（④）の話です。

問3: ベイズワークフローが「一直線ではなく、ぐるぐる回るサイクル」であるのはなぜですか?

現実のデータ分析では、最初のモデルはどこか間違っています。事後予測チェックなどで弱点を見つけたら、モデルを直して①へ戻る（⑦反復）。この「書いて、確かめて、直す」の繰り返しが、ワークフローを循環にしている理由です。bやcのような決まりはありません。

演習

演習1

事前予測チェックと事後予測チェックは、式の上ではよく似ています。両者の式を並べ、どこが違うのか、そしてその違いが何を意味するのかを説明しなさい。

解答を見る

2つの式は次の通りです。

$$ p(\tilde{y}) = \int p(\tilde{y} \mid \theta)\, p(\theta)\, d\theta \qquad p(\tilde{y} \mid y) = \int p(\tilde{y} \mid \theta)\, p(\theta \mid y)\, d\theta $$

違うのは、重みに使う分布です。事前予測では事前分布 \(p(\theta)\)、事後予測では事後分布 \(p(\theta \mid y)\) を使います。つまり「データ \(y\) を見て学んだかどうか」だけが違いです。事前予測は学ぶ前の思い込みからデータを作るので、事前分布の妥当性を点検できます。事後予測は学んだ後のモデルからデータを作るので、モデルが現実を再現できるかを点検できます。

演習2

あるお店の「1日の来客数」をポアソン分布でモデル化しました。事後予測チェックをすると、本物のデータには「来客0人の日」がかなり多いのに、モデルの作る偽データではほとんど0人の日が出ませんでした。このとき、モデル批判としてどんなことが言えますか。考えられる次の一手を1つ挙げなさい。

解答を見る

モデル（ポアソン分布）は「0人の日の多さ」という、データの大事な特徴を取りこぼしている、と批判できます。ポアソン分布だけでは、現実に多い「ゼロ」を十分に作り出せないのです。

次の一手の例: 「ゼロが起きやすい仕組み」を足したモデルに作り直すことが考えられます（たとえば「そもそも休業で0人」と「営業したが偶然0人」を分けて考えるなど）。あるいは曜日や天気といった、来客数を説明する変数を追加することも候補です。いずれにせよ、図1の点線をたどって①へ戻り、モデルを更新して反復します。

※ ここで大事なのは具体的な手法名ではなく、「事後予測チェックでズレを見つけ、原因を考え、モデルを直して戻る」という流れを踏めることです。

演習3

「事前分布を思いきり広くしておけば中立で安全だ」という考え方には、どんな落とし穴がありますか。事前予測チェックの観点から説明しなさい。

解答を見る

事前分布を極端に広く（たとえば \(\mathcal{N}(0, 1000^2)\) のように）すると、事前予測分布が現実にはありえない値を大量に生み出すことがあります。身長の例なら「マイナス3メートル」や「500メートル」のようなデータです。広いこと＝中立、ではありません。むしろ「ありえないデータも等しくありえる」と主張してしまっているのです。

事前予測チェックをすれば、この暴走に推定の前から気づけます。多くの場合、常識的な範囲にゆるく収める弱情報事前分布のほうが安全で、推定も安定します。「無情報なつもりが実は暴れている」という典型的な落とし穴を、事前予測チェックが防いでくれます。

まとめ

  • ベイズワークフローは「モデルを書く → 事前予測チェック → 推定 → 収束確認 → 事後予測チェック → モデル批判・比較 → 反復」という、点検しながら作る一連の流れです。
  • 事前予測チェックは、データを見る前に事前分布だけから仮データを作り、常識的な範囲に収まるかを確かめます。暴れた事前分布を早期に止めます。
  • 事後予測チェックは、事後分布から偽データを作り、本物のデータと見比べます。モデルが現実の「データの形・性質」を再現できているかを見ます。
  • 2つのチェックの式はそっくりで、違いは重みが事前分布か事後分布かの一点だけです。
  • 1つのモデルで終わらせず、批判して作り直す反復こそがワークフローの心臓部。一直線ではなく、ぐるぐる回して育てます。
